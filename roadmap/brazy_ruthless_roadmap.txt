# Brazy.it — ruthless implementation roadmap

This roadmap turns the giant feature blueprint into a build order focused on maximum visible product improvement first, using small iterations and a prioritization mindset based on impact versus effort rather than trying to ship everything at once.[web:293][web:294][web:305] The core idea is to deliver quick wins that immediately improve perceived quality, while planning bigger bets like the layout engine, widget registry, and template ecosystem as structured follow-on milestones.[web:293][web:305]

## Priority model

Each item is grouped into one of four buckets:

- Must build now: core quality blockers or differentiators that shape first impressions.
- Build next: major systems that unlock breadth and competitiveness.
- Build after that: strong feature expansion that deepens the platform.
- Defer for now: useful ideas that should not delay the main leap in quality.[web:305]

The order below assumes one main goal: make the public profile and builder look clearly better, faster, and more complete than what users mentally compare against on guns.lol and haunt.gg.[web:242][web:251][web:262][web:264]

## Phase 0 — standards before features

Before adding more visible features, lock a few standards so the product stops getting messier as it grows.

### 0.1 Design tokens [X]

Build a proper token system first.

Needed tokens:

- Colors: background, surface-1, surface-2, surface-3, border-soft, border-strong, text-primary, text-secondary, text-muted, accent-primary, accent-soft, success, warning, danger.
- Radius: xs, sm, md, lg, xl, 2xl.
- Shadow: low, mid, high, glow-subtle, glow-focus.
- Blur: none, low, mid, high.
- Spacing: 4, 8, 12, 16, 20, 24, 32, 40, 48.
- Typography scale: display-xl, display-lg, heading, subheading, body, caption, micro.
- Motion: fast, base, slow, page, hover.

Why first:

- It stops inconsistent spacing and borders.
- It makes theming much easier later.
- It speeds future redesign work.

### 0.2 Component rules [X]

Define component laws before rebuilding the UI.

Every component should define:

- Default state.
- Hover state.
- Focus state.
- Active state.
- Disabled state.
- Error state if relevant.
- Motion behavior.
- Mobile behavior.

### 0.3 Icon and font system [X]

Pick one main icon system and one secondary brand icon source only, because mixed icon styles instantly lower polish.[web:279][web:283][web:284]

Recommended:

- Lucide or Tabler as the main product icon system.[web:279][web:284][web:286]
- Simple Icons for brands only.[web:283][web:290]
- Google Fonts plus Fontshare as your free type sources.[web:280][web:282]

Core slots:

- Hero surface.
- Identity block.
- Meta row.
- Social/action row.
- Featured widget slot.
- Featured link slot.

Core parts:

- Name + username zone.
- Badge row.
- Bio block.

## Phase 3 — layout engine and the four flagship styles

This is the first big differentiator. Without this, everything feels like one layout with cosmetic changes.

### 3.1 Build a layout registry

Architecture:

- One shared profile schema.
- Layout renderers read the same schema.
- Shared slots render through layout-specific arrangement.

### 3.2 Ship four serious layout families first

#### Minimal

Goal:

- Understated, typography-led, low-glow, sharp.

Needed decisions:

- Narrow width.
- Flat or lightly elevated surfaces.
- Minimal badges.
- Static or very subtle background.

#### Sleek

Goal:

- Premium glass, blur, controlled highlights, glossy but not cheesy.

Needed decisions:

- Strong hero card.
- Integrated socials and music.
- Subtle motion.
- Good surface layering.

#### Modern

Goal:

- Best default for most people; modular, clear, versatile.

Needed decisions:

- Strong section rhythm.
- Clean card system.
- Easy content expansion.

#### Portfolio

Goal:

- More editorial and content-forward.

Needed decisions:

- Wider hero.
- Project-first modules.
- More whitespace.
- Less gamer-style ornamentation.

### 3.3 Add layout previews to builder

Each layout must show a real miniature screenshot-like preview inside the picker, not just a name.

## Quality gate for every release

Before shipping anything, ask:

- Does this make the public profile look better?
- Does it improve first impressions in under 5 seconds?
- Is it visually previewed in the builder?
- Does it fit the design system?
- Does it feel premium without acting premium-gated?
- Is it mobile-safe?
- Is it less corny and more polished than the alternatives?

## Final direction

The winning path is not “add the most features fastest.” The winning path is to first make brazy.it look undeniably premium and visually coherent, then layer on the full free feature set that competitors use as premium bait.[web:293][web:305][web:242][web:254][web:262][web:266] If you do that in this order, users will feel the jump immediately: first the product looks better, then it feels broader, then it becomes obviously better value.[web:293][web:294][web:305]
